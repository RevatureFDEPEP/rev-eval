# Explanation of Newly Added / Modified Files

This document explains why several files appeared in the working tree and why
some existing files were updated. 
## Newly added files

- `.idea/*` (several files)
  - Why they appeared: These are IDE (JetBrains / IntelliJ) project metadata files.
    They are created automatically when the project is opened in the IDE.
  - Why you might keep them: If this repository is intended to be shared with
    a team that uses the same IDE and project configuration, keeping them can
    make onboarding simpler.
  - Why you might remove them: IDE files are usually personal and noisy for a
    shared repository. Consider adding them to `.gitignore` and removing from
    the repo unless the team explicitly wants them versioned.

- `frontend/package-lock.json`
  - Why it appeared: A lockfile generated by `npm` or dependency operations.
    The frontend uses `pnpm` (preferred) which maintains `pnpm-lock.yaml` —
    however developers sometimes run `npm install` locally which creates
    `package-lock.json`.
  - Why you might keep it: Committing lockfiles ensures reproducible installs
    for the package manager that generated them. If your team standardizes on
    `pnpm`, prefer committing `pnpm-lock.yaml` instead and remove `package-lock.json`.
  - Suggested action: Decide on a single lockfile policy (pnpm vs npm) and
    either keep or remove `package-lock.json` accordingly.

## Updated files

- `docker-compose.yml`
  - Why updated: Small improvements to service startup, healthchecks, and
    environment variables to make the stack more reliable for local development
    and for the `api-gateway` to discover backend services via compose DNS.

- `frontend/Dockerfile`
  - Why updated: Standardize multi-stage builds, ensure `pnpm` is installed and
    that production images are smaller and run as a non-root user.

- `services/*/Dockerfile` (question-management-service, test-management-service)
  - Why updated: Add minimal runtime dependencies, healthchecks, and a
    non-root user so containers behave predictably under Docker Compose and
    CI. These changes are operational (runtime) and not related to test code.
